# 🤝 Как помочь проекту

Спасибо, что хотите внести вклад в **MusicForDance**! Этот документ описывает как:

- сообщить об ошибке;
- предложить улучшение;
- прислать pull request с правками.

---

## 🐛 Сообщить об ошибке

1. Откройте [Issues](https://github.com/bivlked/MusicForDance/issues) и проверьте, что аналогичная проблема ещё не описана.
2. Если её нет — нажмите **New issue** → выберите шаблон **🐛 Сообщить об ошибке**.
3. Заполните все поля шаблона (версия `Node.js`, версия `ffmpeg`, операционная система, команда воспроизведения, ожидаемое и фактическое поведение, вывод программы).
4. Если можете, приложите тестовый аудиофайл (короткий — до 10 секунд), на котором проблема воспроизводится. **Не присылайте чужие защищённые авторским правом материалы** — лучше создайте короткий тестовый файл через `ffmpeg`:

   ```bash
   ffmpeg -f lavfi -i "sine=frequency=440:duration=5" -c:a flac test.flac
   ```

---

## 💡 Предложить улучшение

1. Откройте [Issues](https://github.com/bivlked/MusicForDance/issues), выберите шаблон **💡 Предложить улучшение**.
2. Опишите сценарий использования (что Вы хотите делать с инструментом).
3. Опишите, что текущая версия не умеет / делает неудобно.
4. Если есть конкретное предложение по реализации — приложите.

**Прежде чем кодить большую фичу — обсудите её в issue.** Это сэкономит время: возможно, есть нюансы, которые сделают предложенный подход неудачным, или фича уже планируется иначе.

## 📝 Прислать pull request

### Главные правила

1. **Архитектура одного файла** *(single-file)*. Вся логика инструмента должна оставаться в `index.js`. Никаких папок `lib/`, `src/`, никаких пакетов в `node_modules`. Это сознательное ограничение: пользователь должен иметь возможность скопировать **только** `index.js` на флешку и развернуть его на новом компьютере.

2. **Перед PR — прогон smoke-теста.** Если у Вас установлен `ffmpeg`, обязательно запустите:

   ```bash
   node tests/smoke.js
   ```

   Должно вывести `✓ smoke OK (6/6 tests)`. Если хоть один тест упал — разберитесь почему до отправки PR.

3. **Любое утверждение про точность или качество** в `README.md` или `docs/ARCHITECTURE.md` должно быть подкреплено кодом или регрессионным тестом в `tests/smoke.js`. Например, если Вы пишете «теперь поддерживается формат XYZ без потерь» — соответствующий тест должен это подтверждать.

4. **Никаких упоминаний AI-инструментов** в commit-сообщениях, описаниях PR, релиз-нотах или коде. Полный перечень запрещённых строк и команда для проверки — в разделе «Проверка перед коммитом» файла [`docs/STYLE.md`](docs/STYLE.md). Это сознательное правило проекта.

### Стиль коммитов

Простой и понятный: короткое описание (до 72 символов) + при необходимости тело с деталями. Без `Conventional Commits` (`feat:`/`fix:`/...) — это излишне для проекта с одним сопровождающим.

Хорошие примеры:

```text
Поправил снап sample rate для Opus

Раньше Opus сохранял дефолт 48 кГц даже если у источника было 24 кГц.
Теперь функция snapToOpusRate выбирает ближайшее из набора 8/12/16/24/48.
```

```text
README: добавил секцию про русские раскладки путей
```

### Стиль документации

Если правка касается документации (`README.md`, `docs/*.md`, `CHANGELOG.md`, шаблоны) — соблюдайте контракт переводов из [`docs/STYLE.md`](docs/STYLE.md). Главное:

- **На «Вы»** в обращении к пользователю.
- **Без неоправданных англицизмов** — для каждого английского технического термина в [`docs/STYLE.md`](docs/STYLE.md) указан русский эквивалент или правило дублирования при первом упоминании.
- **Внутренние имена фильтров `ffmpeg`** (формат отсчётов, обработка тишины, склейка и пр.) допустимы только в [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md), и там — обязательно с расшифровкой по-русски. Полный список запрещённых терминов — в [`docs/STYLE.md`](docs/STYLE.md).

Перед коммитом запустите две проверки из [`docs/STYLE.md`](docs/STYLE.md) (раздел «Проверка перед коммитом»):

1. Поиск запрещённой технической терминологии в пользовательских разделах.
2. Поиск упоминаний AI-инструментов.

Обе команды должны вернуть пустой результат.

### Изменения в цепочке обработки

Любая правка в функции `buildOutput()`, `processFile()`, `chooseOutputSpec()` или фильтрах `ffmpeg` — **обязательная** прогонка `tests/smoke.js`. Если правка может повлиять на новый сценарий, который текущие 4 теста не покрывают — добавьте новый тест.

### CHANGELOG

Если правка достойна упоминания (новая фича, баг-фикс, заметное изменение поведения) — добавьте строку в секцию `## [Unreleased]` файла [`CHANGELOG.md`](CHANGELOG.md). Формат — [Keep a Changelog 1.1.0](https://keepachangelog.com/ru/1.1.0/).

## 🔧 Локальная разработка

Никаких `npm install` — у проекта нет зависимостей. Никаких сборщиков, транспилеров, минификаторов. `index.js` — итоговый файл, который запускается напрямую.

Клонируйте репозиторий:

```bash
git clone https://github.com/bivlked/MusicForDance.git
```

```bash
cd MusicForDance
```

Проверьте синтаксис основного файла:

```bash
node --check index.js
```

Запустите автоматический smoke-тест (нужен `ffmpeg` в `PATH`):

```bash
node tests/smoke.js
```

Запустите сам инструмент на тестовом файле:

```bash
node index.js "test-track.mp3"
```

Покажите справку по флагам:

```bash
node index.js --help
```
