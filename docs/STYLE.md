# Стиль документации MusicForDance

> ⚠️ **Внутренний документ.** Это контракт для авторов и ревьюеров документации (включая внешних reviewer-ов вроде Codex). Не предназначен для конечных пользователей. Не упоминать в README.

## Назначение

Единый источник правды для терминологии и стиля во всех публичных документах проекта (`README.md`, `docs/ARCHITECTURE.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, шаблоны issue/PR). Если новая правка в любом из этих файлов вводит технический термин — он должен быть согласован с этим документом или добавлен сюда.

## Целевая аудитория публичной документации

1. **Основная**: учитель танцев / музыки / тренер хореографии. Не-программист. На «Вы». Готов скопировать одну команду в терминал, но не готов разбираться что такое sample-rate.
2. **Вторичная**: любопытный пользователь, который хочет понять как работает внутри. Технически грамотный, но всё-таки русскоязычный.
3. **Третичная**: разработчик, который хочет форкнуть / контрибьютить. Этому достаточно `CONTRIBUTING.md` и `docs/ARCHITECTURE.md`.

## Правила перевода

### 1. Полные русификации

Если есть устоявшийся русский термин — используем его, без английского дублирования.

| Английский | Русский |
|------------|---------|
| pitch preservation | без изменения высоты тона *(в пользовательских текстах)* / сохранение высоты тона *(в углублённом)* |
| lossless | без потерь |
| lossy | с потерями |
| pipeline | цепочка обработки |
| encoder | кодировщик |
| decoder | декодировщик |
| fallback | резервный вариант |
| drag-drop / drag and drop | перетащить (мышью) |
| threshold | порог |
| loop / iteration | проход / итерация |
| flag (CLI) | флаг / параметр |
| output / input file | выходной / входной файл |
| temp file / temporary | временный файл |
| crash | сбой / падение |
| zombie process | повисший процесс |
| edge case | пограничный случай |

### 2. Дублирование (русский + английский в скобках)

Термины, у которых русский эквивалент менее узнаваем технической аудиторией. **Только при первом упоминании в документе**, дальше по тексту — короткая форма.

| При первом упоминании | Дальше по тексту |
|------------------------|------------------|
| битрейт (bitrate) | битрейт |
| кодек (codec) | кодек |
| частота дискретизации (sample rate) | частота дискретизации (короткую форму «частота» **не** используем — слишком общее слово; либо полностью, либо «sample rate» в технических разделах) |
| разрядность (bit depth) | разрядность |
| с точностью до отсчёта (sample-accurate) | посэмплово |
| фазовый вокодер (phase vocoder) | фазовый вокодер |
| transient | **не переводим** — оставляем `transient` курсивом, только в `docs/ARCHITECTURE.md`. В пользовательских текстах не упоминаем (можно сказать «короткие резкие звуки — удары, щелчки, начало нот»). |

### 3. Оставить без перевода

- **Имена форматов и стандартов**: `MP3`, `WAV`, `FLAC`, `AAC`, `ALAC`, `Opus`, `Vorbis`, `OGG`, `AC3`.
- **Имена программ и библиотек**: `ffmpeg`, `ffprobe`, `librubberband`, `libmp3lame`, `Node.js`, `node`, `npm`, `npx`, `git`, `GitHub`, `winget`, `Homebrew`, `apt`.
- **Команды и флаги CLI**: `--mp3`, `--wav`, `--ticks`, `--speeds`, `CLI`, `--help` и т.д. Аббревиатуру `CLI` (command-line interface) при первом упоминании пояснить «программа командной строки».
- **Имена файлов**: `index.js`, `run.bat`, `package.json`, `tests/smoke.js`.
- **Единицы измерения**: `Гц`, `кГц`, `dB`, `мс`, `BPM` (удары в минуту). Битрейт — «кб/с» (не «kbps»).
- **Лицензии**: `MIT`.

### 4. Запрещено в пользовательских разделах

Под «пользовательскими разделами» понимаем: `README.md`, `CONTRIBUTING.md`, `CHANGELOG.md` (записи для пользователей), `.github/ISSUE_TEMPLATE/*`, `.github/PULL_REQUEST_TEMPLATE.md`.

Под «углублённым разделом» понимаем `docs/ARCHITECTURE.md` — там перечисленные ниже термины **разрешены**, но обязательно с расшифровкой по-русски в скобках или соседнем абзаце.

Эти термины — технические внутренности `ffmpeg` или сигнальной обработки.

- `silenceremove`, `aformat`, `asetpts`, `concat` — имена ffmpeg-фильтров. В user-секциях не упоминаем.
- `fltp`, `s16`, `s32`, `f32` — имена sample-форматов. В user-секциях говорим «32-битное число с плавающей точкой» или «целое 16-бит».
- `peak window=0`, `detection=peak` — параметры фильтров. Только с пояснением «алгоритм точного поиска тишины».
- `transient` — оставляем английское имя в `ARCHITECTURE.md` (точного русского нет; «переход» / «всплеск» — двусмысленно).

### 5. Стиль обращения

- **К пользователю на «Вы»** (с заглавной буквы — это уважительная форма обращения, не множественное число): «Если Вы видите ошибку…», «Откройте папку, в которой лежит Ваш трек».
- **Глаголы — повелительное наклонение для команд**: «Скопируйте команду», «Откройте папку».
- **От первого лица — нет** в README и `ARCHITECTURE.md`. В `CONTRIBUTING.md` допустимо «Я принимаю PR…», но лучше нейтрально «Принимаются PR…».

### 6. Имена и заголовки

- **Заголовки секций** — без точки в конце: `## Установка`, не `## Установка.`.
- **Эмодзи в заголовках** — умеренно (1 эмодзи на заголовок, не каждый), только для визуальной навигации.
  - 🚀 — быстрый старт
  - ⚙️ — установка / настройки
  - 📦 — требования / зависимости
  - 📥 — скачать / получить
  - ❓ — частые вопросы
  - 📚 — углублённые материалы
  - 🤝 — как помочь / contributing
  - 📜 — лицензия
  - ⚠️ — предупреждение / важно
  - 💡 — подсказка
- **В тексте абзацев — без эмодзи**, кроме примеров вывода CLI (где `⚠`, `✓`, `✗` приходят из самого инструмента).

### 7. Команды и блоки кода

- **Каждая команда — в отдельном fenced-блоке** с указанием языка (` ```bash `, ` ```powershell `, ` ```text ` для вывода).
- **Однострочные команды — без `$` префикса**: пользователь копирует целиком.
- **Многошаговые блоки — отдельные fenced-блоки** для каждого шага с пояснением между ними.
- **Команды для Windows** — указываем `powershell` или `cmd` явно, если синтаксис различается.

### 8. Ссылки

- **Внутренние** — относительные: `[CHANGELOG](CHANGELOG.md)`, `[архитектура](docs/ARCHITECTURE.md)`.
- **На GitHub** — полный URL только когда нельзя относительной (например, ссылки на release).
- **Внешние ресурсы** — открытым URL без сокращений типа bit.ly.

## Проверка перед коммитом

Перед публикацией документации (из `CONTRIBUTING.md`):

```bash
# 1. Запрещённая терминология в пользовательских разделах:
grep -inE '\b(silenceremove|aformat|asetpts|concat|fltp|s16|s32|f32|transient)\b|peak window=0|detection=peak' \
  README.md CONTRIBUTING.md CHANGELOG.md \
  .github/ISSUE_TEMPLATE/*.md .github/PULL_REQUEST_TEMPLATE.md \
  2>/dev/null

# 2. AI-упоминания (исключаем сам STYLE.md — он перечисляет паттерны и матчился бы сам):
grep -irE 'claude|anthropic|chatgpt|gpt-[0-9]|сгенерировано|generated by|🤖' \
  --exclude=STYLE.md *.md docs/ .github/ 2>/dev/null
```

Оба должны вернуть пусто. Замечания:
- **(1)** не сканирует `docs/ARCHITECTURE.md` — там эти термины разрешены с расшифровкой.
- **(2)** исключает `docs/STYLE.md` (этот файл) — он перечисляет паттерны для проверки, в нём они появляются как примеры, а не как упоминания AI-инструментов в публичной коммуникации.
